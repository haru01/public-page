応急処置の失敗

テストなしリリースとリグレッションの悪循環

ループを読み解く

B ループ（短期的な改善）

リリースを急ぎたい → テストを書かずにコードだけ実装する → 開発が速い → すぐリリースできる → ビジネス要求が満たされる

このバランスループは即座に成果を出す。プロダクトオーナーは「もう動いてるの？」と喜び、ステークホルダーも素早い対応を評価する。だからこそチームは「テストは後で書こう」という行動パターンを強化していく。

R ループ（長期的な悪化）

テストを書かずにリリースする → リグレッションに気づけない → 本番障害が増える → 緊急対応に追われる → 開発時間がさらに圧迫される → ますますテストを書く余裕がなくなる → さらにテストなしでリリースする

ここに遅延がある。テストの欠如はすぐには問題にならない。コードベースが小さいうちは頭の中で影響範囲を把握できる。しかし数週間〜数ヶ月かけてコードが成長するにつれ、ある日突然「修正を入れたら別の画面が壊れた」という形で表面化する。

段階的に進行する症状

Stage 1: 快調な出発
  • テストなしでもサクサク機能を出せる
  • コードが小さいから頭の中で把握できる
  • 「テストは後で書けばいい」という楽観
  • リリース速度が速く、チームの評価も高い
Stage 2: 違和感の兆し
  • 修正のたびに予想外の箇所が壊れる（月に1回程度）
  • 「ここ変えて大丈夫かな？」と不安を感じる箇所が出てくる
  • リリース前の手動テストが少しずつ長くなる
  • 「テスト書かなきゃ」と言いつつ、次の機能開発に追われる
Stage 3: 減速と障害対応
  • リリースのたびに手動での確認に半日かかる
  • 本番障害が月に何度も発生し、緊急対応に追われる
  • バグ修正が新たなバグを生む連鎖が起きる
  • 開発時間の大半が障害対応に消える
Stage 4: 恐怖と行き詰まり
  • 誰もコードを触りたがらない。リリースのたびに祈る
  • 「全部書き直してテスト入れよう」という議論が出る
  • ベロシティが初期の数分の一に
  • ビジネス側との信頼関係が崩壊する
Insight

応急処置の失敗に対するレバレッジは、テストを「余裕があるときに書くもの」ではなく、コードを書く行為の一部として組み込むことにある。TDDやテストファーストまでいかなくとも、変更のたびに最低限のリグレッションテストを追加するだけで安全ネットは着実に育つ。テストを書く時間は「コスト」ではなく、将来の障害対応時間を削減する「投資」だ。

問題のすり替わり（Shifting the Burden / Addiction）

if文パッチの中毒——問題のすり替わり

ループを読み解く

B ループ上段（応急処置の誘惑）

バグが報告される → if文で条件分岐を追加して対処する → バグが直る → 問題が「解決した」ように見える

「この条件のときだけ動作がおかしい」と報告が来る。コードを読んで原因を特定し、if (specialCase) を1行足す。テストして直ったことを確認し、プルリクエストを出す。レビューも通る。10分で終わる。だからこそチームは次もこの手段に手を伸ばす。

B ループ下段（根本解決への道）

バグが報告される → データモデルや処理フローを設計し直す → コードの構造が明瞭になる → 例外ケースが自然に処理される → バグが根本から解消される

バグが特定条件でしか起きないのは、データの流れや責務の分割が現実の業務ルールと合っていないからだ。設計を見直せば、例外ケースを特別扱いする必要がそもそもなくなる。しかしこのループには遅延がある。設計の見直しは数スプリントかかり、その間は目に見える成果が出ない。「バグ1件に数週間かけるのか」とステークホルダーに問われれば、if文パッチのほうが「正解」に見える。結果として、根本解決はいつも後回しになる。

R ループ（中毒の構造）

if文パッチを重ねる → 分岐が複雑化し、コードの見通しが悪くなる（副作用） → 次の変更で別の条件を見落とす → 新たなバグが生まれる → またif文パッチで対処する

ここが「中毒（Addiction）」と呼ばれる所以だ。パッチを当てるたびにコードの分岐が増え、全体を把握するのが難しくなる。把握できないからまたバグが出る。バグが出るからまたパッチを当てる。やがてリファクタリングの見積もりは「数ヶ月」に膨らみ、誰も手を出せなくなる。

段階的に進行する症状

Stage 1: 初回のパッチ
  • 「この条件だけ特殊なので」とif文を1つ追加する
  • 10分で修正完了、素早い対応が評価される
  • 根本対応は「次のスプリントで」と先送り
  • コードに // TODO: あとで設計を見直す が1つ増える
Stage 2: 常態化
  • 同じファイルにif文パッチが何度も追加される
  • 「とりあえず分岐で対処」が暗黙のパターンに
  • 新メンバーが「なぜこの条件分岐があるんですか？」と聞いても誰も答えられない
  • リファクタリングチケットがバックログに溜まり続ける
Stage 3: 依存の深化
  • 分岐の上に分岐が積み重なり、ネストが深くなっていく
  • 1つのif文を外すと何が壊れるかわからない
  • バグ修正のプルリクエストが別のバグを生む確率が上がる
  • 「この処理、全部書き直さないと直せません」が増える
Stage 4: 中毒からの脱出困難
  • 設計の見直しに数ヶ月かかると見積もられ、誰も着手できない
  • 「作り直したほうが早い」が本気で議論される
  • チームが疲弊し、「また同じところのバグか」と士気が下がる
  • ビジネス側は「なぜ同じバグが何度も出るのか」を理解できない
Insight

問題のすり替わりに対するレバレッジは、if文パッチを禁止することではない。緊急対応は時に必要だ。鍵は、パッチを打つたびに根本対応をセットで計画することにある。「if文を1つ足したら、設計改善チケットを1つ作る」をルール化し、スプリントごとにリファクタリング枠を確保する。応急処置は借金、設計改善は返済——この認識をチーム全体で共有する。

目標のなし崩し（Drifting Goals / Eroding Goals）

スケジュール遅延による「目標のなし崩し」

ループを読み解く

B ループ上段（目標を下げる誘惑）

当初の品質基準と現状の開発スピードとの間にギャップが生じる → プレッシャーが高まる → 「現実的な目標に見直そう」と目標を下方修正する → ギャップが縮まる → プレッシャーが解消される

このループは即効性がある。品質基準を下げれば、テスト工数は減り、スコープを削ればリリースは早まる。痛みは一瞬で消える。だからこそ、この手段は一度使うと繰り返し使いたくなる。

B ループ下段（現状を引き上げる努力）

ギャップが生じる → プレッシャーが高まる → プロセス改善や技術負債の返済に取り組む → 開発スピードと品質が向上する → ギャップが縮まる

こちらのバランスループには遅延がある。プロセス改善の効果が数値に表れるまで数スプリントかかる。技術負債の返済は短期的には「何も生み出していない」ように見える。ステークホルダーの忍耐が持たず、上段のループが選ばれやすい。

なぜ目標は下がり続けるのか

問題は、上段のループを一度回すと新しい低い目標が次のサイクルの出発点になることだ。次にギャップが生じたとき、チームはすでに「目標を下げる」という成功体験を持っている。こうして品質基準はスプリントごとに少しずつ侵食され、気づいたときには当初の目標とは似ても似つかない水準になっている。

段階的に進行する症状

Stage 1: 最初の妥協
  • リリースが遅れ、「今回だけ」テスト項目を減らす
  • 「重要な機能だけ先に出そう」とスコープを削る
  • チームは「次回は元に戻す」と約束する
  • 小さな妥協は誰の記憶にも残らない
Stage 2: 妥協の常態化
  • 「現実的な目標設定」が美徳として語られ始める
  • テストカバレッジが毎スプリント少しずつ下がる
  • 「Done の定義」がいつの間にか緩くなっている
  • 「前もこれでOKだった」が判断基準になる
Stage 3: 品質の地盤沈下
  • バグが本番環境で頻発するが「想定内」と処理される
  • 新メンバーが低い基準を「普通」だと思って育つ
  • プロセス改善の提案が「理想論」と退けられる
  • 顧客が品質低下に気づき、信頼が揺らぎ始める
Stage 4: ゆでガエル
  • 当初の品質基準を覚えているメンバーがいなくなる
  • 競合との品質差が決定的に開く
  • 「うちはこういう会社だから」と諦めが支配する
  • 改善しようにも、何が正しい基準なのかわからなくなる
Insight

目標のなし崩しに対するレバレッジは、目標を聖域にすることだ。ギャップを感じたとき、目標を下げるのではなく、現状を引き上げるための時間とリソースを確保する。具体的には、品質基準やDoneの定義を明文化し、変更には意思決定者の承認を必須とする。「目標を下げるコスト」を可視化し、チーム全員がその誘惑に自覚的でいられる仕組みを作る。

成長の限界

スタートアップの初期成長と組織の壁

ループを読み解く

R ループ（成長のエンジン）

製品開発と営業に注力する → ユーザー数・売上が伸びる → 成功体験が投資意欲を高める → さらに開発と営業に邁進する

この強化ループが初期の急成長を生む。PMF（Product-Market Fit）を掴んだスタートアップほど、このループが力強く回り、「やればやるほど伸びる」という成功体験がチーム全体を駆動する。

B ループ（見えない天井）

ユーザー数・売上が急増する → 組織のキャパシティが追いつかなくなる → サポート品質の低下・社内の混乱 → ユーザー離脱・評判悪化 → 成長が鈍化する

ここに遅延がある。組織の限界はすぐには見えない。売上が倍になっても、採用・オンボーディング・社内プロセスの整備には数ヶ月かかる。限界が顕在化した頃には、すでに顧客やメンバーの信頼が損なわれている。

段階的に進行する症状

Stage 1: 爆発的な立ち上がり
  • PMFを掴み、ユーザーが急増する
  • 少人数で全てをこなせている
  • 「このまま一気にいける」という高揚感
  • メディアや投資家からの注目が集まる
Stage 2: 綻びの兆候
  • カスタマーサポートの返信が遅れ始める
  • 新メンバーが入っても即戦力にならない
  • 「あの人に聞かないとわからない」が増える
  • 創業メンバーが疲弊し始める
Stage 3: 成長の鈍化
  • 解約率（チャーン）がじわじわ上昇する
  • 採用しても定着せず、離職が相次ぐ
  • 障害やクレームへの対応が後手に回る
  • 「人が足りない」が全部署の口癖になる
Stage 4: 成長の壁
  • 売上の伸びが完全に頭打ちになる
  • 組織内の対立やサイロ化が深刻化する
  • 優秀なメンバーが「この組織の将来が見えない」と去る
  • 創業者が「何のために始めたんだっけ」と立ち止まる
Insight

成長の限界に対するレバレッジは、Rループ（成長の活動）を加速させることではなく、Bループの制約要因を先回りして取り除くことにある。具体的には、組織能力への投資——採用・育成の仕組み化、プロセスの標準化、権限委譲——を成長が鈍化する前に始めることだ。

成長と投資不足（Growth and Underinvestment）

成長と投資不足（完全版：基準の妥協ループ付き）

ループを読み解く

R ループ（成長のエンジン）

積極的なマーケティング活動 → ユーザー数・需要が急増する → 成功体験がさらなる成長投資を呼ぶ → さらにマーケティングを強化する

成長の限界のR ループと同じ構造だ。需要が伸びるほど成長活動に力が入り、さらに需要が伸びる。しかし、需要の増加は同時にサービス能力への負荷を増大させる。

B ループ1（能力不足による成長の制約）

需要が急増する → 実際のサービス品質が低下する → パフォーマンス基準とのギャップが開く → 投資の必要性が認識される → 人材やインフラへの投資が行われる → サービス品質が回復する

ここに致命的な遅延がある。投資を決断してから効果が現れるまで数ヶ月かかる。その間、品質低下は続き、顧客は離れ続ける。しかしこのアーキタイプの本質的な問題は、遅延だけではない。

B ループ2（基準の妥協——最も危険なループ）

サービス品質が低下する → パフォーマンス基準とのギャップが開く → 「この程度は許容範囲だ」と基準そのものを引き下げる → ギャップが見かけ上縮まる → 投資の必要性が感じられなくなる → 投資が行われない → 品質は低いまま定着する

これが「成長と投資不足」を他の原型と区別する核心構造だ。「目標のなし崩し」のB ループと同じメカニズムが、投資判断を内側から蝕む。品質が下がっても「うちはこのレベルでやっていける」と基準を下げてしまえば、投資の危機感は消える。危機感が消えれば投資は行われず、能力は永遠に不足したままだ。

段階的に進行する症状

Stage 1: 順調な成長
  • ユーザー数が毎月倍増する
  • 少数精鋭のチームで回せている
  • 「この調子でもっとアクセルを踏もう」
  • 能力への投資は「まだ早い」と先送りされる
Stage 2: 品質低下と投資の迷い
  • サポートの返信が数日遅れるようになる
  • サーバー障害やパフォーマンス低下が増える
  • 「投資すべきか？」と議論されるが、投資の効果に確信が持てない
  • 「今は成長優先」と投資判断が先送りされる
Stage 3: 基準の妥協
  • 品質低下が長引き、チームが現状に慣れてくる
  • 「まあ、このくらいなら許容範囲だよね」と基準が暗黙に下がる
  • 投資提案に対して「本当に必要？今のままでも回ってる」と返される
  • 顧客も品質に不満を感じつつ、代替がなければ離れない
Stage 4: 成長の完全停止
  • 低い品質が「この会社のレベル」として固定される
  • 競合が高品質で参入し、顧客が一気に流出する
  • 「なぜ成長が止まったのか」を分析するが、原因が能力不足だと気づかない
  • 当初の品質基準を知る人がおらず、何に投資すべきかもわからなくなる
Insight

成長と投資不足に対するレバレッジは2つある。第一に、投資の遅延を織り込んで先行投資すること。問題が見えてからでは構造的に遅い。第二に、そしてより重要なのは、パフォーマンス基準を聖域にすることだ。基準の妥協ループが回り始めると、投資の必要性そのものが消失する。品質基準を明文化し、「現状に合わせて基準を下げる」という選択肢を意思決定の場から排除する。

エスカレーション

社内プラットフォーム競争のエスカレーション

ループを読み解く

B ループ左（チームAの論理）

チームAの社内シェアが下がる → 「自分たちの基盤を強化しなければ」 → チームAが独自ツールを開発 → Aの社内シェアが上がる → チームBのシェアが相対的に下がる

チームAにとっては合理的な行動だ。社内での影響力を維持するために、自前の基盤に機能を追加し、他チームへの採用を推進する。個別に見れば正しい判断に見える。

B ループ右（チームBの論理）

チームAの社内シェアが上がる → チームBが脅威を感じる → 「対抗して自前の基盤を作ろう」 → チームBが独自ツールを開発 → Aの社内シェアが相対的に下がる

チームBもまた合理的に行動している。Aの基盤に依存すれば、自チームの自律性が失われると感じる。だから自前で作る。両者とも合理的に行動しているにもかかわらず、組織全体では非合理な結果が生まれる——これがエスカレーションの本質だ。

段階的に進行する症状

Stage 1: 善意の出発
  • チームAが社内の共通課題を解決するツールを作る
  • 「みんなに使ってほしい」と善意で公開する
  • チームBは「うちのニーズには合わない」と感じる
  • それぞれが「より良いもの」を目指し始める
Stage 2: 対抗意識の芽生え
  • チームBが「自分たちに最適化した」代替ツールを作る
  • 社内で「A派」と「B派」が生まれる
  • 両チームが機能追加で差別化を図り始める
  • 経営層は「競争は健全」と静観する
Stage 3: 軍拡競争
  • 両チームが相手の機能を取り込み、重複が膨らむ
  • 新入社員が「どっちを使えばいいの？」と混乱する
  • ドキュメントとサポートの負担が倍増する
  • 連携が必要な場面で互換性の問題が頻発する
Stage 4: 共倒れ
  • 組織全体の保守コストが許容範囲を超える
  • どちらのツールも中途半端な状態で放置される
  • 「結局どっちもやめて外部サービスを使おう」という議論が出る
  • 両チームのメンバーが疲弊し、モチベーションが低下する
Insight

エスカレーションに対するレバレッジは、どちらかを「勝たせる」ことではなく、競争の構造そのものを変えることにある。早期に共通ゴールと評価基準を定め、「組織全体への貢献」にインセンティブを再設計する。両チームが勝ち負けではなく、役割分担として協調できる枠組みを作ることが鍵だ。

成功の限界（強者はますます強く）

予算獲得における「成功の限界（強者はますます強く）」

ループを読み解く

R ループ左（Aの好循環）

プロジェクトAが成果を出す → 「Aは優秀だ」と評価される → Aに予算・人材が優先的に配分される → Aの成果がさらに向上する

これはマタイ効果（「持てる者はさらに与えられる」）の構造だ。初期の成功が資源を呼び込み、資源がさらなる成功を生む。Aのチームは潤沢なリソースのもとで存分に力を発揮できる。

R ループ右（Bの悪循環）

Aに資源が優先配分される → Bへの配分が減る → Bの成果が低迷する → 「Bはダメだ」と評価される → さらにBから資源を引き上げてAに回す

Bの低迷は能力の問題ではなく、構造の帰結だ。リソースが不足すれば成果が出ず、成果が出なければさらにリソースを削られる。初期にわずかな差しかなかった2つのプロジェクトが、資源配分の偏りによって決定的な格差に変わる。成功も失敗も、個人の能力以上にシステムの構造が生み出している——これが「強者はますます強く」の本質だ。

段階的に進行する症状

Stage 1: わずかな初期差
  • プロジェクトAとBが同時に立ち上がる
  • Aがわずかに先に成果を出す（運や偶然も含む）
  • 経営層が「Aは有望だ」と注目し始める
  • この段階では差はごくわずかで、逆転可能に見える
Stage 2: 資源配分の傾斜
  • Aに優秀なエンジニアが追加配属される
  • Bの予算が「来期に見直し」と削られる
  • Aは潤沢なリソースで加速、Bはリソース不足で停滞
  • Bのメンバーが「不公平だ」と不満を漏らし始める
Stage 3: 格差の固定化
  • Aが社内の「成功事例」として全社に紹介される
  • Bのメンバーがモチベーションを失い、優秀な人がAに異動する
  • Bの遅延・品質低下が「やはりBは駄目だった」と解釈される
  • 資源を均等にすべきという声が「非効率だ」と退けられる
Stage 4: 一方の消滅
  • Bが事実上の凍結・中止に追い込まれる
  • Bが持っていた戦略的価値や知見が組織から失われる
  • Aへの過度な依存がリスクになるが、代替手段がもうない
  • 「最初から全部Aに集中すべきだった」という誤った結論が定着する
Insight

成功の限界に対するレバレッジは、成功しているAを弱めることではなく、資源配分の基準そのものを見直すことにある。過去の実績だけでなく、戦略的重要性・ポテンシャル・リスク分散の観点で配分を決める。また、BがAと同じ土俵で評価される構造自体を疑い、それぞれのプロジェクトに固有の成功基準を設定する。

共有地の悲劇

牧草地の「共有地の悲劇」

ループを読み解く

R ループ左（A社の成長エンジン）

A社が羊を放牧する → 収益が増える → 成功体験が投資意欲を高める → さらに放牧を拡大する

A社にとっては合理的な経営判断だ。牧草地は「ただで使える」共有資源であり、放牧を増やせば増やすほど収益が伸びる。利益を最大化するためには、可能な限り多くの羊を放つのが正解に見える。

R ループ右（B社の成長エンジン）

B社が羊を放牧する → 収益が増える → 「A社に負けるわけにはいかない」 → さらに放牧を拡大する

B社もまったく同じ論理で動いている。A社が放牧を増やしているのを見れば、自社も増やさなければ競争に負ける。個別の合理性が、全体の非合理を生む——これが共有地の悲劇の核心だ。

B ループ（共有資源の枯渇）

A社とB社が放牧を拡大する → 牧草地の負荷が累積する → 牧草地が荒廃する → 牧草が育たなくなり、羊が痩せる → A社・B社ともに収益が低下する

ここに遅延がある。牧草地の劣化はすぐには見えない。今日の過剰放牧の結果が表れるのは数ヶ月後だ。しかもA社から見れば、牧草地の劣化が自社の放牧のせいなのかB社のせいなのか区別できない。「自分だけ控えても、相手が増やしたら損をする」という囚人のジレンマが生まれ、誰も自制できなくなる。

段階的に進行する症状

Stage 1: 豊かな共有地
  • 共有資源（牧草地）は十分にあり、誰も不自由しない
  • 各社が自由に資源を利用し、順調に成長する
  • 資源の管理ルールは存在しないか、緩い
  • 「まだまだ余裕がある」という楽観が支配する
Stage 2: 利用の拡大
  • 各社が成長を追い、資源の利用量が増加する
  • 競合他社の動きが見え、「出遅れたくない」という焦り
  • 資源の劣化の兆しが見え始めるが、因果関係が不明瞭
  • 「自分だけ控えても意味がない」という論理が広まる
Stage 3: 資源の枯渇
  • 共有資源の劣化が目に見えて進行する
  • 収益が頭打ちになり、各社が「もっと使わなければ」と利用を強化
  • 互いに相手の過剰利用を批判するが、自らは控えない
  • 自主的な規制の呼びかけは「不公平だ」と合意に至らない
Stage 4: 共有地の崩壊
  • 共有資源が回復不能なレベルまで劣化する
  • 全社の収益基盤が同時に崩壊する
  • 「なぜもっと早く手を打たなかったのか」と後悔が広がる
  • 資源の再生には、利用停止と長期間の回復期間が必要になる
Insight

共有地の悲劇に対するレバレッジは、個々の行動を制限することではなく、共有資源の管理者（または管理ルール）を設けることにある。利用量の上限設定、利用コストの導入、または資源の状態を全員に可視化する仕組みを作る。「自分だけ控えても損」という構造を、「全員でルールを守れば全員が得をする」という構造に変える。

バラバラの目標

部門間の「バラバラの目標」

ループを読み解く

B ループ左（営業部の論理）

営業部が売上目標を追う → 顧客の要望に応えて特注品を受注する → 製品仕様が多様化する → 製造部が標準化を進められず混乱 → 製品の提供価値が不安定になる → 営業部が「もっと顧客に寄り添わねば」とさらに特注を増やす

営業部にとって、顧客の声に応えることは正義だ。受注を増やし、売上を伸ばすことが自部門の使命であり、KPIもそれを求めている。個別に見れば合理的な行動だ。

B ループ右（製造部の論理）

製造部がコスト削減を追う → 生産ラインの画一化を推進し特注を拒否する → 製品仕様が画一化する → 営業部が顧客のニーズに応えられなくなる → 製品の提供価値が不安定になる → 製造部が「もっと効率化しなければ」とさらに標準化を進める

製造部にとって、コスト効率と品質安定は正義だ。標準化を進め、ムダを排除することが自部門の使命であり、KPIもそれを求めている。これもまた個別に見れば合理的な行動だ。二つの「正義」が共有の状態をめぐって綱引きし、互いの努力を打ち消し合う——これがバラバラの目標の本質だ。

段階的に進行する症状

Stage 1: それぞれの最適化
  • 営業部と製造部がそれぞれの目標を掲げる
  • 両部門とも「自分たちは正しいことをしている」と確信
  • 部門間の接点は少なく、問題は表面化しない
  • 経営層は部門ごとのKPIを別々に管理している
Stage 2: 摩擦の発生
  • 営業が受注した特注品を製造が「対応できない」と拒否
  • 「営業が無理な約束をする」「製造が融通が利かない」と互いに不満
  • 顧客への納期遅延や仕様変更が発生し始める
  • 両部門の間に連絡会議が設けられるが形骸化する
Stage 3: 対立の固定化
  • 両部門が相手の行動を「問題の原因」と認識する
  • 部門間のコミュニケーションが最低限に減る
  • 問題がエスカレーションされ、経営層の仲裁が常態化する
  • 顧客満足度と利益率が同時に低下する
Stage 4: 組織の分断
  • 部門間の対立が「文化」として定着する
  • 有能な人材が板挟みに疲れて離職する
  • 競合他社に対する組織全体の対応力が低下する
  • 「うちの会社は営業と製造の仲が悪い」が社内の常識になる
Insight

バラバラの目標に対するレバレッジは、どちらかの部門を勝たせることではなく、上位の共通目標を設定することにある。「顧客への提供価値の最大化」という全社目標を定め、両部門のKPIをそこから逆算して再設計する。部門横断のプロダクトチームを作り、対立ではなく協調が報われるインセンティブ構造に変える。
