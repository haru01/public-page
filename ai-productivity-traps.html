あなたの組織で、こんなことが起きていませんか

あなたの組織で、こんなことが起きていませんか。

「AIツールを入れたら、個人の開発スピードは確かに上がった。体感で2倍は速くなったと思う。でも——組織全体のデリバリーは、なぜか速くなっていない。」

これは、あなただけの感覚ではありません。

DORA 2024の調査では、AI支援ツールを使った個人の体感速度は+25%向上した一方、組織のデリバリー安定性はむしろ-19%低下しました。CEO・CFOの80%が「AI導入が生産性に影響していない」と回答しています。

杉野和馬さん（LayerX）は、この逆説を「熱狂と齟齬」と呼びました。個人は速くなった。でも組織は速くなっていない。この矛盾の背後には、5つの構造的な罠が隠れています。

一緒に、その構造を見ていきましょう。

Insight

個人が速くなれば組織も速くなる——この直感は、残念ながら正しくありません。個人と組織の間には「接続・調整層」という見えない天井があり、ここを素通りしてAIの効果が組織に届くことはないのです。

罠1: 成長の限界——「個人の速さ」が天井にぶつかる

AI個人生産性の成長と「接続・調整層」の天井

ループを読み解く

R ループ（個人生産性のエンジン）

AIツールを使う → 個人のコード生産量が増える → 「AIすごい！もっと活用しよう」 → さらにAIツールへの投資と習熟が進む → 個人の速度がさらに上がる

このループが最初の「熱狂」を生みます。GitHub Copilotを入れれば、コーディング速度は体感で2倍になる。ChatGPTに設計相談すれば、調査時間が半分になる。個人レベルでは、AIの効果は疑いようがありません。だからこそ、組織はAIツールの導入にアクセルを踏み続けます。

B ループ（見えない天井）

個人の生産量が増える → プルリクエストが増える → レビュー待ちの行列が長くなる → 仕様の確認待ち・意思決定待ちが増える → デリバリーのリードタイムは変わらない（むしろ悪化する） → 「もっとAIを使って個人を速くしなければ」とR ループを加速させる

ここに遅延があります。AIツールを入れた直後は成果が見えます。しかし数ヶ月経つと、バリューストリームの中で「待ち時間」が支配的になっていることに気づきます。杉野さんのスライドが指摘する6層モデルでは、L2（個人生産性）とL4（チーム・プロセス）の間にあるL3（接続・調整層）が最大のボトルネックです。

個人がどれだけ速くコードを書いても、レビュー待ちが3日かかれば、デリバリーは3日遅い。仕様確認に1週間かかれば、AI生成コードの品質は無関係です。個人の「作業時間」をいくら圧縮しても、「待ち時間」が変わらなければ、全体のリードタイムは変わらない。 これが、TOC（制約理論）でいう「非制約の改善は全体を改善しない」という原則そのものですね。

段階的に進行する症状

Stage 1: 熱狂の始まり
  • AIツールを導入し、個人の体感速度が劇的に向上する
  • 「コーディングが2倍速くなった！」という声がチームに広がる
  • 経営層が「全社的にAIを導入しよう」と旗を振る
  • ツール利用率がKPIになり、導入数が成果指標になる
Stage 2: 違和感の兆し
  • プルリクエストの数が増えるが、マージまでの時間が延びる
  • 「コードは書けたけど、レビューが回ってこない」という声が出始める
  • AIが生成したコードの品質にばらつきがあり、レビュー負荷が上がる
  • 「速く作れるけど、速く届けられない」というギャップに気づく人が出る
Stage 3: 成長の鈍化
  • デリバリーのリードタイムが改善しないことが数値で見え始める
  • 「AIを使っているのになぜ遅いのか」と経営層が疑問を持つ
  • チームが「もっとAIを使えばいい」と間違った方向に力を入れる
  • バリューストリーム上の「待ち時間」が全体の70%以上を占める
Stage 4: 熱狂の失望
  • 「AI導入は効果がなかった」という結論に飛びつく
  • ツールへの投資が縮小され、使える人と使えない人の格差だけが残る
  • 本質的なボトルネック（接続・調整層）は手つかずのまま
  • 次の「銀の弾丸」を探し始める
Insight

あなたの組織が今 Stage 2〜3 にいるなら、それは「AIが使えない」のではありません。AIの効果が「接続・調整層」という天井にぶつかっているだけです。天井の存在に気づいたこと自体が、次の一手を打つための重要な一歩です。

罠2: 問題のすり替わり——AIツールという「対症療法」への依存

AIツール導入という「問題のすり替わり」

ループを読み解く

B ループ上段（AIツールという即効薬）

組織のデリバリーが遅い → 「AIツールを入れれば速くなる」 → AIツールを全社導入する → 個人の作業速度が上がる → 「生産性が上がった（気がする）」 → 問題が解決したように見える

このループは魅力的です。AIツールの導入は比較的簡単で、効果も即座に体感できる。経営層への報告も「ツール利用率90%達成」と明快です。だからこそ、組織はこの手段に繰り返し手を伸ばすのです。

B ループ下段（組織基礎体力という地道な道）

組織のデリバリーが遅い → CI/CDの整備、プロセス設計の見直し、チーム構造の再編、計測基盤の構築に取り組む → 組織のデリバリー能力が根本的に向上する → 安定したデリバリーが実現する

杉野さんが示した「組織の基礎体力」には、自動化（CI/CD、テスト自動化、IaC）、プロセス設計（アジャイル、WIP制限）、組織設計（チーム構造、知識共有）、計測（DORA Metrics、バリューストリームマッピング）が含まれます。これらは効果が出るまでに数四半期から数年かかります。しかも、「プロセスを改善しました」は「AIを入れました」ほどわかりやすい成果報告になりません。

だからこそ、根本解決はいつも後回しにされます。

R ループ（増幅の構造——最も危険なループ）

AIツールに依存する → 組織の基礎体力を鍛える動機が失われる → 基盤の弱い組織にAIが重なる → AIが既存の問題を増幅する → コードレビューの品質低下、技術負債の加速、チーム間の分断が深まる → デリバリーがさらに遅くなる → 「もっとAIツールを入れなければ」

DORA 2025が指摘した「AIはミラーとマルチプライヤー」という洞察は、まさにこの構造を言い当てています。Booking.comのようにDORA計測×リーンなプロセス×成熟したCI/CDの上にAIを重ねた組織は+16%のスループット向上を実現しました。一方、基盤の薄い組織がAIを入れると、大量の低品質コードが高速に生成され、レビュー負荷と技術負債が加速する——AIが「問題を増幅する鏡」として機能してしまうのです。

段階的に進行する症状

Stage 1: 「AIで解決」の期待
  • 「AIツールを入れれば生産性が上がる」という期待が組織に広がる
  • PoC（概念実証）の段階では劇的な効果が見える
  • 地道なプロセス改善の予算がAIツール導入に振り替えられる
  • 「まずAIを入れてから、その他は後で」という優先順位が暗黙に決まる
Stage 2: 依存の常態化
  • AIツールの利用率がKPIになり、導入そのものが目的化する
  • 「AIを使っているのにデリバリーが遅い」という矛盾が見え始める
  • 「もっと高機能なAIツールを入れれば解決する」と次のツールを探す
  • 組織基盤の弱さは「AIがもっと進化すれば解決する」と先送りされる
Stage 3: 増幅された問題
  • AIが生成する大量のコードがレビューのボトルネックを深刻化させる
  • テスト自動化が未整備のままAI生成コードが本番に入り、品質事故が増える
  • チーム間の調整コストが増大し、「AIのせいで忙しくなった」という声が出る
  • DORA Metricsを計測していないので、何が悪化したのか特定できない
Stage 4: 「AIは使えない」という誤った結論
  • 「結局AIは生産性に貢献しない」という結論に組織が飛びつく
  • AIツールへの投資が打ち切られる
  • 組織の基礎体力は相変わらず弱いまま
  • 次のバズワード（「エージェント型AI」「自律型開発」）に飛びつくサイクルが始まる
Insight

ここで大事なことは、AIツール導入が「悪い」わけではないということです。問題は、AIツールを「代わりに」入れるのか、組織基盤の上に「重ねて」入れるのか——その順序と組み合わせなのです。

罠3: 成長と投資不足——AI時代の「基準のなし崩し」

AI活用の期待成長と組織能力への投資不足

ループを読み解く

R ループ（AI導入の期待エンジン）

AIへの期待が高まる → AIツールの導入を加速する → 「導入しました」という報告が増える → 社内外の注目が集まる → さらなるAI投資への期待が高まる

このループ自体は悪くありません。AI技術への期待が投資を生み、投資が新しい可能性を開く。問題は、このエンジンが回るスピードに、組織の受け入れ能力が追いつかないことです。

B ループ1（組織能力の制約）

AI活用が進む → 組織の基盤（プロセス、計測、チーム構造）が追いつかない → 実際のデリバリー成果が期待を下回る → 期待基準とのギャップが開く → 投資の必要性が認識される → プロセス設計や計測基盤に投資される → 遅延を経て組織能力が向上する → AI活用の成果が改善する

ここに致命的な遅延があります。CI/CDの整備、DORA Metricsの導入、チーム構造の見直し——これらは効果が出るまでに半年から1年かかります。しかも「プロセスを改善しています」は、「新しいAIツールを入れました」ほど経営層にインパクトのある報告になりません。投資判断が先送りされやすい構造があるのです。

B ループ2（基準のなし崩し——最も危険なループ）

AI活用の成果が期待を下回る → 「AIの効果は個人の体感速度で測ればいい」と基準を下げる → ギャップが見かけ上縮まる → 「組織基盤への投資は不要」という結論に至る → 投資が行われない → 組織能力は弱いまま固定される

ここが杉野さんの指摘する核心です。スライドの6層モデルで言えば、L1（ツール導入量）をKPIにしてしまうと、L3〜L5の問題が見えなくなるのです。「ツール利用率90%」「個人の体感速度+50%」——これらの数字は良い。でも「組織のデリバリー安定性は-19%」という現実は、計測していなければ見えません。

計測していなければ基準を下げていることにすら気づかない。気づかなければ投資の必要性を感じない。感じなければ投資されない。これが「成長と投資不足」の最も恐ろしい点です。

段階的に進行する症状

Stage 1: 期待と熱狂
  • 「AI導入で生産性2倍」という目標が掲げられる
  • ツール導入予算が大幅に確保される
  • 組織基盤への投資は「AIの効果を見てから」と先送りされる
  • 個人の体感速度が向上し、「順調だ」という報告が上がる
Stage 2: 成果のギャップと投資の迷い
  • 組織全体のデリバリー速度が改善しないことに気づき始める
  • 「まだツールの習熟が足りない」「もっと良いツールがある」と説明される
  • 組織基盤への投資が提案されるが、「効果が不確実」と承認されにくい
  • AIツールの追加導入で問題を解決しようとするサイクルが始まる
Stage 3: 基準のなし崩し
  • 「AI生産性は個人レベルで測定すればよい」と基準が暗黙に変更される
  • DORA Metricsやバリューストリームマッピングの導入は「過剰な計測」と見なされる
  • 「ツール利用率」が成果KPIとして定着する
  • 組織デリバリーの遅さは「AIとは別の問題」として切り離される
Stage 4: 構造的な停滞
  • 「AI導入は成功した（個人は速くなった）」と報告されるが、事業成果は変わらない
  • 組織基盤は弱いまま、AIによって既存問題が増幅され続ける
  • 「AI生産性」を正確に測る基準を誰も持っていない
  • 競合他社（Booking.comのように基盤とAIを両輪で回した組織）に決定的な差をつけられる
Insight

あなたの組織が「AIは個人レベルでは効いている。組織レベルは別の話だ」と言い始めていたら、それは Stage 3 のサインかもしれません。基準が下がっていることに気づいたこと自体が、転換点になりえます。

罠4: 成功の限界——AI格差が組織を分断する

AI活用格差の固定化——「強者はますます強く」

ループを読み解く

R ループ左（AIリーダーの好循環）

あるチームが早期にAIを使いこなす → 生産性の高さが目に見える成果になる → 「あのチームはすごい」と社内で評価される → 優秀なエンジニアが「あのチームで働きたい」と集まる → AI活用のノウハウがさらに蓄積する → 成果がさらに向上する

最初のきっかけは、ほんの小さな差です。たまたまAIに詳しいメンバーがいた。たまたまチームリーダーが新しいツールに前向きだった。しかし「成功の限界（強者はますます強く）」の恐ろしいところは、この小さな初期差が、リソース配分の偏りを通じて、決定的な格差に変わっていくことです。

R ループ右（取り残されたチームの悪循環）

AIリーダーチームにリソースが集中する → それ以外のチームへのAI投資・支援が減る → AI活用が進まず、旧来の方法で開発を続ける → 成果が相対的に低迷する → 「あのチームはAIに向いていない」と評価される → さらにリソースが削られる

ここで重要なのは、取り残されたチームの低迷は能力の問題ではなく、構造の帰結だということです。同じエンジニアでも、AIツールの導入支援があり、使い方を学ぶ時間があり、成功体験を積める環境があれば、活用は進みます。しかし「あのチームに任せよう」という一言で、その環境が奪われてしまう。

杉野さんのスライドが指摘する「実装×個人に投入が集中」という現象は、組織内でも同じように起きています。AI活用が得意な個人・チームにさらなる投入が集中し、それ以外は「まだ早い」と後回しにされる。部分最適化が全体最適を阻む——これは6層モデルのL2（個人）とL4（チーム）の間でも再現されている構造なのです。

段階的に進行する症状

Stage 1: わずかな初期差
  • 一部のチーム・個人が自発的にAIツールを使い始める
  • 早期導入者が目覚ましい生産性向上を見せる
  • 経営層が「AIの成功事例」として社内に紹介する
  • 他のチームは「うちもそのうち」と静観する
Stage 2: リソース配分の傾斜
  • AI活用チームに追加予算・新しいツール・優秀な人材が配分される
  • 「まずは成果の出ているチームに集中投資しよう」という合理的判断
  • 他のチームのAI導入支援は「次のフェーズで」と先送りされる
  • AI活用チームと非活用チームの間で「壁」が生まれ始める
Stage 3: 格差の固定化
  • AI活用チームが社内の「エリート」と見なされるようになる
  • 非活用チームのメンバーが「自分たちは取り残された」と感じ、士気が下がる
  • 優秀な人材がAI活用チームへの異動を希望し、格差がさらに拡大する
  • 組織全体のAI成熟度は、最も遅いチームの水準に引きずられる
Stage 4: 組織の分断
  • 「AI組」と「非AI組」という暗黙のカーストが定着する
  • AI活用チームへの過度な依存がリスクになる（キーパーソン依存）
  • 非活用チームのプロダクトやサービスの品質が相対的に低下する
  • 「全員でAIを使う」という当初の目標を誰も覚えていない
Insight

あなたの組織で「AIが得意な人に任せよう」という言葉が頻繁に聞こえるなら、それは Stage 2〜3 のサインかもしれません。一部の「AIヒーロー」に依存する構造は、短期的には効率的に見えますが、長期的には組織全体のAI活用能力の天井を作ってしまいます。

罠5: バラバラの目標——「速く作る」と「正しいものを作る」の綱引き

開発効率とプロダクト価値の「バラバラの目標」

ループを読み解く

B ループ左（開発チームの論理——速さの追求）

開発チームが「AIで生産性を上げる」というKPIを追う → AIを活用してコード生産量・機能リリース数を最大化する → 大量の機能が短期間で実装される → しかし、顧客にとって価値のある機能かどうかは未検証 → 使われない機能が増え、プロダクトが複雑化する → デリバリーの質が低下する

杉野さんの6層モデルの最上位、L6（事業価値層）が突きつける問いはこれです——「速く届ける ≠ 正しいものを届ける」。AIによってコーディング速度が2倍になったとき、2倍の機能を作るのが正解でしょうか。それとも、同じ量の機能を作りながら、浮いた時間で「そもそもこれは作るべきか」を検証するのが正解でしょうか。

開発チームにとって、AIで加速した生産性を「目に見える成果」にする最も簡単な方法は、機能の数を増やすことです。PRの数、リリースした機能数、閉じたチケットの数。しかしこれらはすべて「活動量」であり、「価値」ではありません。

B ループ右（プロダクト/事業側の論理——正しさの追求）

プロダクト/事業側が「正しいものを届ける」というKPIを追う → 顧客検証・仮説検証・ユーザーインタビューに時間をかける → 作るべきものが精査される → しかし、検証に時間がかかりリリースが遅れる → 開発チームが「もっと速く決めてほしい」と不満を持つ → デリバリーの量が低下する

プロダクト側もまた合理的に行動しています。間違ったものを作るよりも、正しいものを慎重に見極める方がいい。しかし開発チームがAIで加速しているのに、仕様の決定速度が変わらなければ、「仕様待ち」が新たなボトルネックになります。二つの「正しさ」が、デリバリーの質と量というシーソーの上で綱引きしている——これがバラバラの目標の構造です。

杉野さんが指摘した「効率と有効性は独立した変数」という洞察は、まさにこの構造を言い当てています。効率（速く作る能力）をいくら上げても、有効性（正しいものを作る能力）が同時に上がるわけではない。AIは効率を劇的に改善したが、有効性には手をつけていない。その結果、効率と有効性のギャップがかつてないほど開いているのです。

段階的に進行する症状

Stage 1: それぞれの正義
  • 開発チームは「AIでこんなに速く作れるようになった」と誇る
  • プロダクト側は「何を作るかが大事」と主張する
  • 両者のKPIは別々に設定され、接点は少ない
  • 「速さ」と「正しさ」の矛盾は表面化していない
Stage 2: 摩擦の発生
  • 開発チームが「仕様が来ない。手が空いている」と訴える
  • プロダクト側が「こんなに速く作られても検証が追いつかない」と困惑する
  • AIで量産された機能の利用率データを見ると、使われていない機能が増えている
  • 「速く作ったのに価値が出ていない」という矛盾が数字に表れ始める
Stage 3: 対立の固定化
  • 開発チームは「プロダクト側が遅い」、プロダクト側は「開発が勝手に作る」と互いを批判
  • 手戻り（作ったけど使われない機能のリワーク）が増え、実質的な生産性が低下する
  • 経営層が「AI導入したのに事業成果が出ない」と疑問を持つ
  • 「もっとAIを使って速く作れ」vs「もっと慎重に検証しろ」の対立が深まる
Stage 4: 高速な無駄の量産
  • AIで加速された開発力が「間違ったものの高速量産」に使われている
  • プロダクトの複雑性が爆発し、保守コストが膨れ上がる
  • 顧客が「機能は増えたけど、欲しいものがない」と離反し始める
  • 「たくさん作ったのに、なぜ事業が成長しないのか」という問いに誰も答えられない
Insight

AIが「速さ」を解決した今、最も価値のある時間の使い方は「もっと速く作ること」ではなく、「何を作るべきかを正確に見極めること」かもしれません。速く作れるからこそ、立ち止まって考える余裕が生まれた——そう捉え直すことが、この罠から抜け出す第一歩です。

なぜこうなるのか——5つの罠が絡み合う構造

ここまで5つの構造を別々に見てきましたが、実は、これらは互いに強め合う関係にあります。

罠1→罠2: 「成長の限界」で個人の速さが天井にぶつかると、組織は「もっとAIツールを」と「問題のすり替わり」のループに入ります。 罠2→罠3: すり替わりの結果、組織基盤への投資が先送りされ、「成長と投資不足」が進行します。投資不足で成果が出ないと基準が下がり、基準が下がるとさらに投資の必要性が見えなくなる。 罠1→罠4: 個人の速さで成果が出たチームにリソースが集中し、「成功の限界」が発動します。AI格差が固定化されると、組織全体のAI成熟度はいつまでも上がらない。 罠1→罠5: AIで「速さ」だけが加速すると、「正しさ」の検証が追いつかず、「バラバラの目標」が顕在化します。速く作れるようになったのに事業成果が出ない——L6（事業価値層）の課題はここに根ざしています。

5つの罠が連鎖するとき、組織は「AIは使えない」という誤った結論に向かうか、あるいは「AIは一部のエリートの個人ツール」という矮小化された理解に留まり続けます。

でも、これは誰のせいでもありません。構造がそうさせているのです。個人が速くなれば組織も速くなるはずだ、という直感はごく自然なもの。AIツールに期待するのも当然のこと。問題は直感の誤りではなく、組織というシステムが持つ非線形性——部分の改善が全体の改善に直結しないという性質——を見落としていることにあります。

レバレッジポイント——どこを変えれば流れが変わるか

ところで、「レバレッジポイント」という考え方をご存じですか。

システムの中には、小さな力で大きな変化を生み出せるポイントがあります。

力ずくでシステムを変えようとするのではなく、構造の急所に働きかけるのです。

あなたの組織では、次の5つのポイントが鍵になります。

レバレッジ 1: 計測基準を「個人の速さ」から「組織のフロー」に変える

  • 対象ループ: 成長と投資不足の B ループ2（基準のなし崩し）
  • なぜここが急所なのか: 計測基準が変われば、何に投資すべきかの判断が変わります。「AIツール利用率」ではなく「デリバリーのリードタイム」「変更失敗率」「デプロイ頻度」（DORA Metrics）を基準にすれば、L3（接続・調整層）のボトルネックが可視化され、投資の優先順位が構造的に変わります
  • 期待される変化: 基準が組織レベルに設定されることで、「個人が速くなった」だけでは満足せず、「組織として速くなったか」を問う文化が生まれます

レバレッジ 2: AIの投資先を「実装×個人」から「接続・調整層」へシフトする

  • 対象ループ: 成長の限界の B ループ（接続・調整層の天井）
  • なぜここが急所なのか: 杉野さんの2軸モデル（開発ライフサイクル×スコープ）が示す通り、現在のAI投入は「実装×個人」に集中しています。しかし最大のボトルネックはL3（レビュー、仕様確認、意思決定）です。AIの力を「コードを書く」から「チーム間の接続を滑らかにする」方向に振り向ければ、制約要因そのものが緩和されます
  • 期待される変化: 「待ち時間」が短縮され、個人の速さがそのまま組織の速さに転換される割合が高まります

レバレッジ 3: AIとセットで「組織の基礎体力」に投資する

  • 対象ループ: 問題のすり替わりの R ループ（既存課題の増幅）
  • なぜここが急所なのか: DORA 2025が示した「AIはミラーとマルチプライヤー」は、AIが基盤の強い組織でのみ正の効果を生むことを意味します。Booking.comの事例（DORA計測×リーンプロセス×成熟CI/CD×AI）は、複数の投入面を掛け合わせたときに初めて相乗効果が生まれることを実証しています
  • 期待される変化: AIが「問題を増幅する鏡」ではなく「強みを増幅する鏡」として機能し始めます

レバレッジ 4: AI活用の知見を「属人化」から「組織の仕組み」に変える

  • 対象ループ: 成功の限界の R ループ左右（格差の拡大）
  • なぜここが急所なのか: AI活用の成果が一部のチーム・個人に閉じている限り、組織全体のAI成熟度は上がりません。成功チームの知見を他チームに移転する仕組み——ペアプログラミング、社内勉強会、プロンプトライブラリの共有、AI活用ガイドライン——を作ることで、リソース配分の偏りという構造そのものを変えられます
  • 期待される変化: 「AIが得意な一部の人」から「組織全体がAIを活用できる」へ。底上げが進むことで、L3（接続・調整層）の改善にもAIが使われ始めます

レバレッジ 5: AIで浮いた時間を「もっと作る」ではなく「何を作るかの検証」に使う

  • 対象ループ: バラバラの目標の B ループ左右（速さと正しさの綱引き）
  • なぜここが急所なのか: AIが開発の「速さ」を解決した今、最大のボトルネックは「正しいものを見極める力」に移っています。速さで浮いた時間をプロトタイピング・顧客検証・A/Bテストに再投資すれば、「速く作る」と「正しいものを作る」が対立ではなく相互補完の関係に変わります。杉野さんの6層モデルでいうL6（事業価値層）に直接効くレバレッジです
  • 期待される変化: 「速く作ったのに使われない」が減り、作ったものの価値実現率が上がる。開発チームとプロダクト側の目標が揃い始めます
Insight

つまり、こういうことです。AIツールの導入は「入口」であって「ゴール」ではありません。入口の先にある「計測基準の設計」「接続・調整層への投資」「組織基盤との掛け合わせ」「知見の組織的な共有」「浮いた時間の戦略的再投資」——この5つに同時に取り組んだとき、初めてAIは組織を速くします。

明日からできること——アクション案

大きな改革は必要ありません。まずは小さな一歩から始めましょう。

アクション 1: バリューストリームの「待ち時間」を可視化する

  • 誰が: 開発チームのリード、またはスクラムマスター
  • 何を: 1つのフィーチャーが「着手」から「本番リリース」に至るまでの時間を計測し、「作業時間」と「待ち時間」に分解する
  • いつ: 今週中に1つのフィーチャーで試す
  • どうやって: 直近のリリースを1つ選び、各工程の開始・終了時刻を追跡する。Jira/Linear等のタイムスタンプから「レビュー待ち」「仕様確認待ち」「デプロイ待ち」を抽出する。作業時間と待ち時間の比率を円グラフにして、チームに共有する
  • 気をつけること: これは誰かを責めるための計測ではありません。「構造を見るための計測」であることを最初に明言してください。数字が予想以上に悪くても、それは構造が見えたということ。良い兆候です

アクション 2: DORA Metricsの4指標を週次で追跡し始める

  • 誰が: エンジニアリングマネージャーまたは開発基盤チーム
  • 何を: デプロイ頻度・リードタイム・変更失敗率・復旧時間の4指標を定義し、ダッシュボードを作成する
  • いつ: 次の2週間で初期ダッシュボードを作り、4週間後に最初のトレンドレビューを行う
  • どうやって: まずは手動集計でもよい。CI/CDのログからデプロイ頻度とリードタイムを、障害チケットから変更失敗率と復旧時間を追跡する。完璧を求めず、「まず計測を始める」ことを優先する
  • 気をつけること: DORA Metricsを「評価」に使うと、数字を操作する動機が生まれます。あくまで「改善のための鏡」として位置づけてください。また、AI導入前のベースラインがなければ、今から取り始めれば今がベースラインになります

アクション 3: AI投資とセットで「接続・調整」の改善テーマを1つ選ぶ

  • 誰が: 経営層・VPoE・開発リード
  • 何を: 次のAIツール投資の予算のうち、20%を「接続・調整層の改善」に充てる。例えば、AIコードレビュー支援、AI要件定義支援、意思決定の非同期化ツールなど
  • いつ: 次の四半期計画で組み込む
  • どうやって: 杉野さんの2軸モデル（開発ライフサイクル×スコープ）を使い、現在のAI投資が「実装×個人」に偏っていないか棚卸しする。バリューストリーム分析で特定した最大の「待ち時間」に対応するAI活用テーマを1つ選ぶ
  • 気をつけること: 「接続・調整」の改善は技術だけでは解決しません。会議体の見直し、意思決定プロセスの短縮、チーム構造の再設計など、組織設計の要素が含まれます。技術チームだけでなく、マネジメント層の巻き込みが不可欠です

アクション 4: AI活用の「成功事例」を仕組みとして組織に展開する

  • 誰が: AI活用が進んでいるチームのリード + エンジニアリングマネージャー
  • 何を: AI活用の知見を属人的な「あの人がすごい」から組織の共有資産に変える。具体的には、プロンプトライブラリの整備、AI活用のペアプログラミングセッション、週次の「AI Tips共有会」の開催
  • いつ: 来週から、まずは月2回の共有会を試す
  • どうやって: AI活用チームのメンバーが、非活用チームと1対1でペアプロする「AI Buddy制度」を始める。30分でいい。「自分がAIをどう使っているか」を見せるだけで、学びのハードルは劇的に下がる。成功パターンはプロンプトテンプレートとしてNotionやConfluenceに蓄積する
  • 気をつけること: 「教える側」と「教わる側」という上下関係にならないよう注意。AI活用チームも他チームから学ぶことがある（ドメイン知識、顧客理解など）。双方向の学び合いとして設計する

アクション 5: AIで浮いた時間の20%を「何を作るか」の検証に再投資する

  • 誰が: プロダクトマネージャー + 開発チームリード
  • 何を: AIによって短縮された開発時間の一部を、プロトタイピング・顧客インタビュー・A/Bテストに充てる
  • いつ: 次のスプリントから、各スプリントに「検証タイム」を1日確保する
  • どうやって: スプリントプランニングで「作る機能」と「検証する仮説」をセットで計画する。AIで高速にプロトタイプを作り、顧客に見せてフィードバックを得る。「作る前に確かめる」サイクルを回す。作らなくてよいものが1つ見つかれば、それだけで数スプリント分のコストが浮く
  • 気をつけること: 「検証に時間を使う＝遅くなる」という誤解が生まれやすい。実際には、間違ったものを作る手戻りの方がはるかにコストが高い。「速く作れるからこそ、何を作るかに時間を使える」というフレーミングが大切です

おわりに——構造に気づいたあなたへ

ここまで読んでくださって、ありがとうございます。

今回の分析で見えてきたのは、「AIが使えない」のでも、「あなたの組織が悪い」のでもないということです。個人の速さが組織の速さに変換されない構造がある——それだけのことです。

5つの構造的な罠。成長の限界、問題のすり替わり、成長と投資不足、成功の限界、バラバラの目標。どれも、個人が合理的に行動した結果として生まれる罠です。誰も悪循環を望んでいなかった。でも、構造が見えなければ、同じパターンを繰り返してしまう。

杉野さんが示した「診断→構造化→介入点→レバレッジ設計」というプロセスは、まさにシステム思考そのものです。計測して、構造を見て、急所に働きかける。

全部を一度に変えようとしなくて大丈夫です。

まずはバリューストリームの「待ち時間」を1つだけ可視化してみてください。

そこに見えた数字が、次の一歩を教えてくれます。

小さな変化が、やがてシステム全体の流れを変えていきます。

Insight

AIは組織の基礎体力を映す鏡であり、増幅器です。鏡に映った姿が気に入らないとき、鏡を取り替えても何も変わりません。変えるべきは、鏡の前に立っている自分たち——組織の構造そのものです。
